Un problema è detto **calcolabile** se è possibile risolverlo algoritmicamente. Possiamo formalizzare il problema con una funzione matematica che avrà ovviamente input e output. Gli input saranno il dominio della funzione mentre l'output il codominio. TUTTE le funzioni che vedremo avranno input e output con **numeri naturali (ovvero numeri da 0 a infinito).**

# Funzioni totali e parziali

![[Pasted image 20251013134114.png]]
In sintesi, le funzioni totali sono funzioni che ti assicurano che per QUALSIASI input ci sarà un output mentre quelle parziali no. Un esempio di funzione totale è ad esempio la funzione di somma, mentre una funzione parziale tipica potrebbe essere la divisione che non è definita per 7 / 0.

Le **funzioni totali** rappresentano dunque un **sottoinsieme** delle **funzioni parziali**. A volte potremmo sacrificare la generalità del problema per garantirne la calcolabilità.

Per definire dunque il concetto di calcolabilità di una funzione, è necessario enunciare la tesi di **Church-Turing**

Tesi:
	"**Qualsiasi funzione che si possa determinare tramite qualche procedura algoritmica è parzialmente calcolabile**."

# Linguaggio S

Il linguaggio S è un linguaggio formale per la scrittura di algoritmi utilizzabile per la verifica della calcolabilità di un problema. Questo linguaggio infatti è minimale nelle istruzioni, possedendo solo quelle eseguibili da qualsiasi compilatore e con le quali è possibile, per quanto in molte più linee, ricostruire un qualsiasi algoritmo.

## Sintassi

### Variabili

**Input**: Si indicano SEMPRE con delle X il cui pedice
**Output**: Sempre indicata come Y, non può mai esserci più di una Y nel programma come "dichiarazione" 
**Temporanee**: Si indicano SEMPRE con Z. Di default il valore è 0

Il pedice delle lettere che indicano variabili di input / temporanee, può essere un numero naturale qualsiasi. Quando il pedice delle variabili di Input / Temporanee non è specificato allora significa che il pedice è 1. 

### Etichette
Le etichette alfabetiche (label) servono per indicare specifiche linee su cui poter effettuare dei salti (jump). Vengono utilizzate le lettere dell’alfabeto dalla A alla D numerate, con il caso speciale dell’etichetta E, usata per la terminazione del programma, in generale quando si indica il salto ad un'etichetta che non esiste allora il programma esce.

### Statement
Il linguaggio S contempla solo **4 tipi di statement**, o asserzioni, che possono essere combinate o meno a etichette per formare istruzioni. Nello specifico questi statement sono:

* **Pigra: V<--V**: non fa nulla 
* **Incremento: V<--V+1**, incrementa una variabile qualsiasi V di 1; 
* **Decremento: V<--V−1**, decrementa una variabile qualsiasi V di 1, se essa contiene il valore 0 l’istruzione viene ignorata; 
* **Salto: IF V != 0 GOTO L**, passa alla prima linea etichettata dall’etichetta generica L se la variabile generica V è diversa da 0, altrimenti l’istruzione viene ignorata. Nel caso l’etichetta L non esista, il programma termina. 

```

1 [A] X <-- X − 1 
2 Y <-- Y + 1 
3 IF X !=0 GOTO A

Questo programma emula un ciclo for che fa in modo di trasferire il valore di X in Y. Se ad esempio f(4) = 4, f(5) = 5 ecc... Nel caso in cui X = 0 allora ritorna 1

```

# Caratterizzazione di un S-Programma

Chiamiamo **S-Programma** una qualsiasi **lista finita d’istruzioni in linguaggio S**. Una volta definito formalmente il concetto di S-Programma possiamo, usando le proprietà che derivano da questa definizione, andarlo a caratterizzare fino a cogliere il parallelismo suddetto tra S-Programmi e funzioni calcolabili. Innanzitutto, per descrivere un S-Programma è importante tenere traccia dell’aggiornamento delle sue variabili. Per fare ciò ci avvaliamo del concetto di Stato di un Programma.

## Stato di un Programma

**Definizione di Stato**
![[Pasted image 20251013142129.png]]

Oltre alla definizione di Stato, intuitivamente V sarà la variabile DEL PROGRAMMA (e non una a caso) mentre n sarà il valore in quell'istante.

Lo **stato però non è sufficiente a descrivere un programma**, in quanto non tiene traccia dell’**ordine** di esecuzione delle istruzioni. Per avere una descrizione puntuale di un programma di **’l’ istruzioni** , è però necessario contestualizzare il suo stato in un determinato momento dell’esecuzione. Per fare questo ci avvaliamo del concetto **d’istantanea, o snapshot**.

```

1 [A] IF X!!=0 GOTO B 
2 Z <-- Z + 1 
3 IF Z !!= 0 GOTO E 
4 [B] X <-- X − 1 
5 Y <-- Y + 1 
6 Z <-- Z + 1

```

Ok ora creiamo l'insieme di tutti gli stati di questo programma (Σ) il cui stato iniziale è { X = 2, Y = 0}.

Σ = {X = 2, Y = 0}, {X = 2, Y = 0}, {X = 1, Y = 0}, {X = 1, Y = 1}, {X = 1, Y = 1}, {X = 0, Y = 1}, {X = 0, Y = 2}

**Definizione di istantanea**: ![[Pasted image 20251013141255.png]]

Dove con l + 1 codifichiamo la terminazione del programma, essendo un’istruzione con tale indice non esistente. Chiamiamo l’istantanea con questo indice terminale. Le istantanee permettono in oltre di descrivere consequenzialmente le istantanee successive di un programma se abbinate alla istruzione indicizzata da i, a patto che essa non sia un’istantanea terminale.

![[Pasted image 20251013143436.png]]

Il **calcolo terminale** altro non è che la lista delle istantanee che portano poi alla terminazione del programma.

Esempio di calcolo terminale:
0. Stato Iniziale: σ1 = {X = 2, Y = 0} 
1. Istantanea 1: (1, σ1); 
2. Istantanea 2: (2, σ2) con σ2 = {X = 1, Y = 0}; 
3. Istantanea 3: (3, σ3) con σ3 = {X = 1, Y = 1}; 
4. Istantanea 4: (1, σ3); 
5. Istantanea 5: (2, σ4) con σ4 = {X = 0, Y = 1}; 
6. Istantanea 6: (3, σ5) con σ5 = {X = 0, Y = 2};
7. Istantanea 7: (4, σ5) terminale; 
8. Stato Finale: σ4 = {X = 0, Y = 2};

	È possibile notare che se venissero forniti più input di quelli necessari, la successione d’istantanee rimarrebbe invariata, ignorando gli input in eccesso. Se ad esempio prendiamo come input la n-upla (2, 3, 4), lo stato iniziale sarà σ1 ∪ {X1 = 3, X2 = 4}, ma si comporterà come se fosse stato fornito solo 2 come input. Inoltre è possibile generalizzare il calcolo ulteriormente, contemplando anche la possibilità d’input sottodimensionati rispetto al numero di variabili richieste. Considerando infatti le variabili d’input senza inizializzazione esplicita come poste a 0, l’avanzamento delle istantanee procederà normalmente fino alla terminazione in maniera analoga a quelle appena viste.
