The structure of an operating system (OS) is a complex interplay of several key components, each serving specific roles to ensure efficient and reliable operation. Here's a detailed breakdown:

1. **Kernel**: The central component responsible for managing processes, handling interrupts, and providing essential services. It can be monolithic (all functions in one place) or microkernel-based (splitting responsibilities), affecting flexibility and efficiency.

2. **System Calls**: These are the API through which applications communicate with the kernel, enabling tasks like file operations and network communication. They differ from library functions by being direct calls to the OS.

3. **Device Drivers**: Translate high-level OS commands into low-level hardware operations, allowing interaction with devices like GPUs and network cards. Some drivers run in kernel space for efficiency, while others operate in user space for better security.

4. **User Space vs Kernel Space**: Kernel space handles critical tasks directly interacting with hardware, while user space isolates applications to prevent crashes from affecting the entire system.

5. **Processes and Threads**: A process is a running program instance, containing memory and resources. Threads are execution paths within a process, managed by the OS for CPU allocation and scheduling, using algorithms that balance fairness, efficiency, and priority.

6. **Interprocess Communication (IPC)**: Facilitates communication between processes via methods like pipes, sockets, shared memory, or message queues, crucial for applications needing to interact across different services.

7. **Shell**: The interface through which users interact with the OS, executing commands either via command-line interfaces (e.g., bash) or graphical environments. It relies on init processes and service managers.

8. **System Initialization**: Init process starts during boot, setting up essential services and processes. This ensures all necessary daemons and services are operational before user interaction begins.

9. **Daemons**: Background processes providing system services like web servers (Apache), databases (MySQL), and network services (SSH). They run continuously to handle incoming requests.

10. **Boot Process**: Begins with BIOS/UEFI loading the OS, followed by kernel initialization of hardware and essential systems, then user-space processes启动 via init.

11. **Real-Time Systems**: Hard real-time systems guarantee task completion deadlines (e.g., aerospace), while soft real-time offers good performance without strict guarantees, suitable for general use.

12. **Design Trade-offs**: Balancing ease of use for end-users with flexibility and customization needs for system administrators, affecting OS design priorities.

In summary, the OS structure involves a kernel providing foundational services, device drivers enabling hardware interaction, processes managing application execution, daemons offering continuous services, and init orchestrating startup. IPC allows communication between user and kernel spaces, ensuring efficient operation and reliable service delivery.

## 1. Il processo e le sue necessità fondamentali

Un **processo** è un **programma in esecuzione**.

In un sistema **multiprogrammato** il SO deve gestire più processi contemporaneamente → compito principale: **schedulazione** (decide quali processi mandare in esecuzione e quando).

Per essere eseguito un processo necessita di:

- essere **caricato in memoria principale**
- accedere a **risorse** (file, dispositivi, memoria, …) tramite **chiamate di sistema**

Alla **terminazione** il SO deve occuparsi del **rilascio e recupero** delle risorse allocate.

### Processi single-thread vs multi-thread

- **Single-thread** → una sola sequenza di istruzioni
- **Multi-thread** → più flussi di esecuzione (**thread**) all’interno dello stesso processo
  - → più **program counter** da gestire
  - → ulteriore livello di parallelismo / interleaving

Nei sistemi moderni → molti processi + molti thread + esecuzione su **più core**  
→ lo scheduler deve assegnare thread/processi ai vari core (esecuzione **contemporanea** possibile)

## 2. Principali funzionalità offerte dal SO per i processi

Il SO mette a disposizione primitive (chiamate di sistema) per:

- **Creare** processi
- **Lanciare** un programma (es. exec family)
- **Sospendere / riprendere** esecuzione
- **Sincronizzare** processi
- **Far comunicare** processi tra loro (IPC)

Queste primitive saranno studiate sia **teoricamente** che **praticamente** (programmazione concorrente in **C** con API **POSIX**).

> [!example] Esempi classici di primitive POSIX (anticipate)
> - `fork()` → duplica (clona) il processo corrente → padre e figlio identici da quel punto in poi
> - `wait()` → sincronizzazione: il padre si mette in attesa della terminazione del figlio (rendez-vous)
> - `exec*()` → il processo “dirotta” la propria esecuzione su un nuovo programma (il corpo del programma cambia, ma resta lo stesso processo)

> [!important] Attenzione
> Fare un ciclo while con fork() senza controllo → creazione esponenziale di processi → esaurimento risorse → blocco macchina

## 3. Gestione della memoria

Per la **multiprogrammazione** → più processi devono coesistere in **memoria principale**.

Non si può caricare tutto subito → caricamento dinamico / parziale.

### Concetti chiave anticipati

- **Memoria paginata** → page fault → trap → il SO va a caricare la pagina mancante da memoria secondaria
- Separazione **memoria logica** (vista dal processo / programmatore) vs **memoria fisica**
- Memoria logica → organizzazione più semplice e lineare
- Meccanismi per ridurre **frammentazione** (problema dei vecchi sistemi con allocazione contigua)

Il SO gestisce anche:

- Allocazione / deallocazione memoria secondaria
- Illusione di memoria principale più grande di quella reale (virtual memory)

## 4. Gerarchia delle memorie e caching

> [!abstract] Gerarchia tipica (dal più veloce al più lento)
> - Registri
> - Cache (L1, L2, L3)
> - Memoria centrale (RAM)
> - SSD
> - HDD
> - Memoria terziaria (archiviazione a lungo termine: tape, cloud, …)

Problemi:

- Differenza enorme nei **tempi di accesso** (da ps a ms / s)
- Differenza in **banda** di trasferimento

Soluzione principale: **caching** + mantenimento **coerenza** (write-back / write-through)

## 5. File system (anticipazione)

Il SO fornisce una rappresentazione **logica uniforme** delle memorie secondarie → astrazione **file**.

Gestisce:

- Creazione / cancellazione file e directory
- Lettura / scrittura
- Permessi
- Strutture dati interne (i-node, directory entry, …)
- Backup

Tutte queste operazioni → **chiamate di sistema** (open, read, write, close, …)

## 6. System Call – Livelli di astrazione

Esempio flusso reale per un comando banale come `cp file.txt output.txt`:

programma utente (es. cp scritto in C)  
↓  
invoca funzioni di libreria standard (es. fopen, fread, fwrite, fclose)  
↓  
funzioni di libreria → invocano **POSIX API** (open, read, write, close, …)  
↓  
POSIX API → contengono istruzioni assembly che preparano registri e fanno **syscall** (es. numero 1 = write su x86_64)  
↓  
**trap** → kernel → vettore interrupt → routine di servizio corrispondente al numero

### Passaggio parametri alle system call (livello macchina)

Tre modalità principali (le prime due sono le più usate oggi):

1. Parametri caricati direttamente nei **registri** (più veloce)
2. Un registro contiene un **puntatore** a una zona di memoria con i parametri
3. (meno usata oggi) parametri nello **stack** utente → kernel li legge (vecchi sistemi)

Nei sistemi moderni → **libc wrapper** copia i parametri dallo stack ai registri **prima** del trap → kernel trova tutto già pronto.

> [!quote] “Più i dati stanno nei registri, più veloce è l’operazione.”

## 7. POSIX vs WinAPI (confronto rapido)

- **POSIX** → standard per sistemi Unix-like (Linux, macOS, …)
  - più pulite, parsimoniose, didattiche
- **WinAPI** → native Windows
  - più parametri, più complesse, meno intuitive

Stesso concetto → interfacce diverse.

## Riepilogo finale

> [!summary] Punti chiave della lezione
> - Un **processo** è un programma in esecuzione; i moderni SO gestiscono **molti processi + molti thread** su **più core**
> - Le **system call** sono il meccanismo fondamentale di comunicazione tra user-space e kernel
> - Esistono diversi **livelli di astrazione**: programma utente → libreria C → POSIX API → syscall (numero + registri) → kernel
> - La **memoria** è gerarchica; il SO crea **illusione** di memoria più grande e gestisce **page fault**, **frammentazione**, **caching**
> - Importante separazione **politica** (cosa vogliamo fare) vs **meccanismo** (come lo facciamo)
