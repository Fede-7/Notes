The text offers an overview of operating system (OS) concepts, focusing on interaction mechanisms, boot processes, architectural designs, and system services. Here's a structured summary:

1. **Interaction Mechanisms**:
   - **System Calls**: Functions allowing user programs to access kernel services.
   - **Interrupts**: Signals from hardware/software that halt execution to handle critical events.

2. **Booting Process**:
   - Initiated by firmware (BIOS/UEFI), which loads the bootloader (e.g., GRUB).
   - Bootloader initializes the kernel and user space, with init systems like systemd launching all processes, including services and user sessions.

3. **OS Architectures**:
   - **Monolithic**: All components in one place (e.g., Linux).
   - **Microkernel-based**: Components run outside the kernel (e.g., macOS).
   - **Real-Time OSes**: Provide strict timing guarantees for critical applications like avionics.

4. **System Services**:
   - Kernel manages hardware drivers, file systems, and network stacks.
   - Userland includes tools like shells, compilers, and libraries, with daemons running as background services (e.g., web servers).

5. **Design Goals**:
   - Balances usability (ease of use, reliability) for users against system-side goals (maintainability, efficiency).
   - Real-time systems prioritize strict timing over user-friendliness.

This overview highlights the OS's structure and design considerations, emphasizing trade-offs between different architectures and priorities. Further exploration into monolithic vs microkernel architectures and real-time systems would enhance understanding.

# Lezione 1 — Struttura e Progettazione di Sistema Operativo

## 🌱 Introduzione al Sistema Operativo (OS)

### Processi Utente e Kernel
> [!important] **Concetto Fondamentale**  
Il sistema operativo organizza i processi in modo gerarchico, partendo dal primo processo utente. Questo approccio permette una gestione ordinata e flessibile dei servizi.

- Il `process daemon` è il primo processo utente.
- I processi kernel lavorano in modo simile agli altri processi ma con priorità specifiche.
- L'albero dei processi è gerarchico, con il processo base come "matrice" per tutti gli altri.

### Modalità di Esecuzione
> [!note] **Modalità Utente vs Sistema**  
Processi utente e sistema coesistono, offrendo flessibilità e parallelismo. I processi utenti possono eseguire funzioni di amministrazione, garantendo un ambiente di lavoro più ordinato.

## 🛠️ Progettazione del Sistema Operativo

### Obiettivi principali
> [!theorem] **Obiettivi chiave**  
I requisiti per un sistema operativo variano tra comodità dell'utente e flessibilità dei sistemisti:

| **Utente**               | **Sistemista**             |
|---------------------------|----------------------------|
| Comodo, affidabile, veloce | Facile da mantenere, efficiente |

### Sistema Operativo a Tempo Reale
> [!example] **Esempio: Waxwork**  
Un sistema operativo con garanzie `hard real-time` sacrificerà comodità per garantire priorità ai processi critici.

- **Garanzie di tempo**: Serve per applicazioni embedded e mission-critical.
- **Impatti sulle prestazioni**: Maggiore flessibilità a discapito delle risposte immediate all'utente.

### Configurazione del Kernel
> [!code] **Esempio di Configurazione**  
Il kernel può essere personalizzato per diverse configurazioni, ad esempio:

```c
// Esempio di moduli abilitati/disabilitati
CONFIG_RT=y  # Abilita supporto a tempo reale
```

## 💡 Politica vs Meccanismo

### Differenza fondamentale
> [!important] **Separazione di Competenze**  
- **Politica**: Decisioni su cosa fare (es. regolazione del time slice).
- **Meccanismo**: Implementazione delle decisioni (es. timer hardware).

### Esempio: Gestione della Memoria
> [!tip] **Trade-off nelle pagine di memoria**  
- **Dimensione delle pagine**: Grande flessibilità vs overhead maggiore.
- **Configurazione**: Politica regola il meccanismo (es. TLB, paging).

## 📦 Evoluzione dei Linguaggi nel Sistema Operativo

### Livelli di Astrazione
> [!info] **Evoluzione del Codice**  
I linguaggi di alto livello (C, C++) rendono più facile il porting tra piattaforme.

- **Linguaggi bassi**: Assembly per strati vicini all'hardware.
- **Linguaggi alti**: Facilitano la portabilità e riducono il vincolo alla piattaforma specifica.

### Concetti di Base
> [!success] **Concetto di Virtualizzazione**  
La virtualizzazione emerge come strumento per disaccoppiare i livelli del sistema operativo, garantendo flessibilità e isolamento.
