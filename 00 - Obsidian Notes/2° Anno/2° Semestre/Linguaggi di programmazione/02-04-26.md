# Sicurezza e controllo di tipo — Lezione: Concetti fondamentali di sicurezza in Java
**Docente:** Non specificato | **Data:** 01-04-2026

## Argomenti trattati
- Definizioni formali di sicurezza tramite verifica bytecode.
- Importanza della tipizzazione forte in Java.
- Gestione automatica della memoria e prevenzione di errori comuni come buffer overflow e memory leakage.

## Corpo della lezione

### Definizioni formali
> [!abstract] Definizione: Sicurezza tramite verifica bytecode
Java utilizza un processo rigoroso per la verifica del bytecode prima dell'esecuzione, garantendo che il codice non violi regole di sicurezza. Questo include controlli su accessi alla memoria, operazioni su stack e conversioni di tipo illegali.

> [!abstract] Definizione: Tipizzazione forte
Java è un linguaggio fortemente tipizzato che impedisce conversioni di tipo illegali e riduce gli errori in fase di compilazione, migliorando la sicurezza del codice.

> [!abstract] Definizione: Buffer overflow e memory leakage
La gestione automatica della memoria da parte della JVM (Java Virtual Machine) previene il consumo eccessivo di risorse e crash del sistema. Questo meccanismo riduce il rischio di errori come memory leakage e accessi non autorizzati a indirizzi di memoria.

### Dimostrazioni presenti
- **Verifica bytecode come meccanismo di sicurezza**: Il processo di verifica del bytecode include controlli su accessi alla memoria, operazioni su stack e conversioni di tipo. Questi controlli sono eseguiti in tempo di esecuzione, non solo in fase di compilazione, per gestire casi indecidibili come gli overflow dei buffer.
  ```mermaid
  flowchart TD
    A[Verifica bytecode] --> B[Controlli accessi memoria]
    B --> C[Controlli operazioni stack]
    C --> D[Controlli conversioni tipo]
    D --> E[Esecuzione in tempo di esecuzione]
    E --> F[Casi indecidibili]
    F --> G[Controlli runtime per indici vettori]
  ```

- **Confronto tra tipizzazione forte e debole**: La tipizzazione forte di Java riduce gli errori in fase di compilazione, mentre la tipizzazione debole ne aumenta. Questo è evidenziato nel confronto tra linguaggi come C e Java.
```mermaid
  xychart
    title Confronto tra tipizzazione forte e debole
    x: Tipizzazione forte (Java)
    y: Errori in fase di compilazione
    data: C (alto), Java (basso)
```

### Spiegazioni del "perché" presenti
- **Java impedisce buffer overflow e memory leakage**: La tipizzazione forte e la verifica del bytecode evitano l'accesso a memoria non autorizzata. La gestione automatica della memoria da parte della JVM (garbage collector) previene il memory leakage, mentre la verifica del bytecode impedisce l'esecuzione di codice non sicuro come attacchi di tipo buffer overflow o abusi di memoria.
- **Aritmetica dei puntatori non è supportata**: I puntatori non sono modificabili, riducendo il rischio di errori di gestione della memoria. Questo design elimina la possibilità di accessi non autorizzati a indirizzi di memoria.

- "Costa. Va fatto però una tantum quando carichi la classe. Quello che si guadagna da un punto di vista di robustezza e sicurezza è impagabile è un costo che vale la pena di sopportare."
  - La verifica del bytecode, pur richiedendo un overhead computazionale, è essenziale per garantire la sicurezza del sistema e prevenire attacchi esterni. Questo costo è giustificato dal rischio di vulnerabilità senza controlli.

### Riepilogo della lezione
> [!summary] Punti chiave della lezione
> - La sicurezza in Java è garantita dalla verifica del bytecode, che include controlli su accessi alla memoria e operazioni su stack.
> - La tipizzazione forte di Java riduce gli errori in fase di compilazione e migliora la sicurezza del codice.
> - Il garbage collector della JVM gestisce automaticamente la memoria, prevenendo errori comuni come memory leakage.

# Compilazione e esecuzione  

## Definizioni formali
> [!abstract] Compilazione in bytecode
Processo che converte il codice Java (`.java`) in un file bytecode (`.class`) tramite `javac`.

> [!abstract] Esecuzione di bytecode
Interpretazione del bytecode da parte della **macchina virtuale Java (JVM)** tramite il comando `java`.

### Dimostrazioni presenti
- **Flusso di lavoro Java**:  
  1. Compilazione in bytecode (`javac`)  
  2. Esecuzione del bytecode (JVM)  
  3. Verifica del bytecode  
  4. Gestione della memoria (Garbage Collector)
  ```mermaid
  flowchart TD
    A[Compilazione in bytecode] --> B[Esecuzione bytecode]
    B --> C[Verifica bytecode]
    C --> D[Gestione memoria]
  ```

### Spiegazioni del "perché" presenti
- **Portabilità del codice**: Il bytecode è platform-independent, il che significa che può essere eseguito su qualsiasi sistema con JVM.

### Avvertenze o affermazioni forti del prof
- "Non è necessariamente la stessa piattaforma su cui gira la macchina virtuale"
  - Indica che il bytecode compilato può essere eseguito su diverse piattaforme, ma non sempre sulla stessa piattaforma su cui è stato originariamente compilato.

### Riepilogo della lezione
> [!summary] Punti chiave della lezione
> - La JVM interpreta il bytecode e lo esegue, fornendo portabilità del codice Java tra diverse piattaforme.
> - Il flusso di lavoro Java comprende la compilazione in bytecode, l'esecuzione tramite JVM e la verifica del bytecode per garantire sicurezza e robustezza.

# Garbage Collector  

## Definizioni formali
> [!abstract] Garbage collector (GC)
Un componente software che gestisce l'allocazione e il rilascio automatico della memoria in un sistema.

> [!abstract] Recupero automatico della memoria
Processo che identifica e libera la memoria non più utilizzata da oggetti non riferibili, evitando il memory leakage.

### Dimostrazioni presenti
- **Funzionamento del GC in parallelo al codice principale**  
  Il garbage collector non viene invocato esplicitamente, funziona indipendentemente, in totale autonomia. Diverse realizzazioni della macchina virtuale possono usare diversi algoritmi di garbage collection.

### Spiegazioni del "perché" presenti
- **Spiegare il meccanismo di gestione della memoria**  
  Il recupero di memoria non più utilizzata viene fatto automaticamente dal linguaggio stesso, dalla JVM, che è un'altra delle componenti che qui non viene evidenziato, però fa parte della macchina.

### Avvertenze o affermazioni forti del prof
- "Il garbage collector non viene invocato esplicitamente, funziona indipendentemente"  
  Indica che il GC opera in modo autonomo e non richiede interventi manuali durante l'esecuzione del programma.

### Riepilogo della lezione
> [!summary] Punti chiave della lezione
> - Il garbage collector è un componente software che recupera automaticamente la memoria non più utilizzata, evitando errori comuni come memory leakage e garantendo una gestione efficiente della memoria.

# Confronto con C e Python  

## Definizioni formali
> [!abstract] Garbage collector vs gestione manuale (C)
Il garbage collector di Java è un componente software che gestisce l'allocazione e il rilascio automatico della memoria, mentre in C la gestione della memoria deve essere fatta manualmente.

### Dimostrazioni presenti
- **Confronto tra linguaggi (passo per passo con scelta di n0)**  
  Confronto tra Java e C sulla base del loro approccio alla gestione della memoria: Java utilizza un garbage collector, mentre C richiede una gestione manuale.

### Spiegazioni del "perché" presenti
- **Mostrare differenze tra linguaggi**  
  La gestione automatica della memoria da parte di Java con il garbage collector offre vantaggi in termini di sicurezza e robustezza rispetto alla gestione manuale, che può portare a errori come memory leakage.

### Avvertenze o affermazioni forti del prof
- "Il Python ha il problema del garbage collector, ma non è un problema intrinseco"  
  Indica che il problema non deriva dal meccanismo stesso del garbage collector, ma dalla specifica implementazione di Python.

### Riepilogo della lezione
> [!summary] Punti chiave della lezione
> - La gestione automatica della memoria tramite garbage collection in Java è un aspetto fondamentale per la sicurezza e la robustezza del sistema, rispetto alla gestione manuale impiegata nel linguaggio C.
