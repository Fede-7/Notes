


  
  
The structure of an operating system can be understood by breaking it down into its core components and their interactions:

1. **Kernel**: The central component responsible for managing processes, handling interrupts, and providing essential services. It acts as a bridge between hardware and software.

2. **Userland**: This layer consists of user-space programs and libraries that run on top of the kernel. These include shells, utilities, and application binaries, which communicate with the kernel through system calls or libraries like glibc.

3. **Init System**: Manages the boot process and starts all other services and applications. Examples include systemd in many Linux distributions, which replaces older systems like sysvinit.

4. **Daemons**: Background processes that provide various services. Examples are cron for task scheduling and sshd for SSH connections. These run continuously to handle requests or events.

5. **Isolation Techniques**:
   - **Chroot Environments**: Provide file system isolation, allowing each process to see a separate root directory.
   - **Containers**: Use chroot along with other namespaces (e.g., PID, mount, and network) to isolate processes more comprehensively, sharing the host OS kernel.

6. **Kernel Types**:
   - **Monolithic Kernels**: Combine all services into one large kernel for efficiency but can be harder to maintain.
   - **Microkernels**: Separate services into user space, enhancing security and reliability by isolating bugs from critical system functions.

7. **Operating System Types**:
   - **Real-Time OSes**: Prioritize timely responses, crucial for applications like aviation or medical equipment.
   - **General-Purpose OSes**: Designed for versatility and user-friendliness, such as Linux, which balances performance across various tasks.

8. **Modularity and Design Principles**: Emphasize separation of concerns and modularity to enhance maintainability and extendibility, with microkernels exemplifying this approach by isolating services in user space.

Understanding these components and their interactions provides a foundational grasp of how an operating system is structured and functions.

# Alcuni concetti fondamentali sui sistemi operativi

## Processi e modalità di esecuzione

### Il processo iniziale (init daemon)
- Il primo processo in Linux è il **processo utente** numero 1, noto come `init daemon`.
- Questo processo non è un processo del kernel ma opera in modalità utente.
- L'insieme dei processi forma un albero di processi che avvia il sistema operativo.

### Modalità utente vs kernel
- I processi utente possono eseguire funzioni di sistema.
- Il kernel lavora in modo separato, con processi di sistema che funzionano come altri processi ma con priorità specifiche.

## Sistema operativo e real-time
### Garanzie di tempo
- Alcuni sistemi operativi offrono garanzie **soft** o **hard** Real-Time.
- Esempi: Waxwork, un sistema orientato a task critici.

### Trade-off nella progettazione
- Sistemi general-purpose (es. Linux) sono adatti per multitasking e interazione utente.
- Sistemi specializzati sacrificano comodità per garantire prestazioni specifiche.

## Configurazione del kernel
### Kernel personalizzato
- I server aziendali spesso utilizzano versioni modificate del kernel (es. supporto Real-Time).
- Il kernel è configurabile e estendibile attraverso moduli.

## Politica vs Meccanismo
### Definizione
- **Politica**: Decisioni su cosa fare (es. regolazione del timer).
- **Meccanismo**: Implementazione delle decisioni (es. strumenti per regolare il timer).

### Esempi
- Gestione della memoria: dimensione delle pagine.
- Paginazione: meccanismo fisico vs logico.

## Progettazione e portabilità
### Evoluzione dei sistemi operativi
- Passaggio da linguaggi di basso livello (assembly) a linguaggi di alto livello (es. C, C++).
- Porting tra piattaforme: vantaggi di linguaggi di alto livello.

### Virtualizzazione e disaccoppiamento
- Utilizzo di emulazione e virtualizzazione per separare strati bassi e alti.

## Note importanti
> [!note] I processi utente e kernel hanno ruoli distinti ma collaborativi nello svolgimento delle funzioni del sistema operativo.
> [!note] La scelta tra un sistema general-purpose o specializzato dipende dagli obiettivi specifici dell'applicazione.
